# P0-002 Worklog: Design tokens and UI primitives

## Status

Completed.

## Goal

Turn the supplied design tokens into a reproducible UI foundation and provide accessible primitives before building product screens.

## Intended files

- `packages/ui/scripts/generate-tokens.mjs` and committed generated TypeScript/CSS token artifacts.
- `packages/ui/src`: Button, IconButton, Panel, Badge, Tooltip, Dialog, Tabs, Progress, Card, and shared styles.
- `apps/web`: a `/components` demonstration route that exercises every primitive.
- Focused UI tests and component-demo screenshot support.

## Constraints

- `design-tokens.json` remains the palette and metric source of truth.
- Raw project palette values may appear only in token sources/generated output or documented visual assets.
- Every action remains keyboard accessible with a visible focus state.
- Reduced motion and high contrast must work at the CSS-foundation level.
- Do not implement product-domain screens or Phaser behavior in this task.

## Acceptance criteria

- Tokens can be regenerated deterministically from `design-tokens.json`.
- Button, IconButton, Panel, Badge, Tooltip, Dialog, Tabs, Progress, and Card are exported and demonstrated.
- Basic semantic and keyboard accessibility tests pass.
- The demonstration uses the package palette and editorial surface direction at 1440 × 900.
- Repository typecheck, lint, tests, and build pass.

## Validation commands

```text
pnpm --filter @signal-atlas/ui generate:tokens
git diff --exit-code packages/ui/src/generated
pnpm typecheck
pnpm lint
pnpm test
pnpm build
```

The browser test harness and a reviewed 1440 × 900 component-demo baseline are included in this task.

## Dependencies and risks

- Token generation must preserve CSS-valid values such as RGBA strings and quoted font stacks.
- Dialog and Tabs require careful keyboard/focus behavior without introducing a heavyweight component framework.
- Tooltip content must not be the only accessible label for an action.
