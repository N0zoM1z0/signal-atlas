# P5-002 Worklog: MCP discovery and connection diagnostics

## Status

Complete.

## Goal

Connect the Pref Gateway to the inspected hosted Preference MCP deployment through the official Streamable HTTP client, discover its usable primitive surface without binding gameplay code to provider names, validate a versioned capability map, and expose safe connection controls and diagnostics to the application.

## Pre-implementation evidence

- The active Pref registration uses Streamable HTTP at the hosted `/mcp` endpoint and authenticates with OAuth-backed bearer credentials.
- The local Preference deployment source pins `@modelcontextprotocol/client` and `@modelcontextprotocol/server` at `2.0.0-alpha.3`; Signal Atlas will align with that exact client version for protocol compatibility.
- The public front door uses `search_tools` for catalog lookup and `call_tool` for provider execution.
- Resource descriptors are available through `list_resources`; prompt descriptors are included by the read-only `onboard` helper when native MCP resource/prompt capabilities are absent.
- Exact catalog lookup confirmed `weather.get_current_conditions` on `weather_toolkit`, with a strict `location` input and read-only/idempotent/non-destructive annotations.
- A source-search candidate was rejected because its current security metadata reports an external-write side effect.
- The integration and credential boundary is recorded in `docs/adr/0001-pref-mcp-integration-boundary.md` before implementation begins.

- `packages/pref-gateway/package.json` and lockfile: official MCP client dependency aligned with the inspected server.

- `apps/orchestrator/src/app.ts` and tests: connection status and explicit connect/disconnect/test endpoints with safe errors.

- Playwright coverage and snapshots as required by rendered diagnostics changes.

## Acceptance criteria

- Streamable HTTP is the only enabled live transport and uses the official MCP SDK.
- Tools, resources, resource templates, and prompts can be listed through native protocol capabilities or the documented Pref helper fallback.
- The checked-in capability map validates before use and the selected provider contract validates against exact live catalog discovery.
- Unknown or unlisted hosts, canonical capabilities, direct tools, and provider tool refs fail closed.
- The frontend can see disconnected, connecting, connected, authentication-required, and error states and can explicitly test/reconnect or disconnect.
- Browser responses contain only bounded primitive summaries and fixed safe errors; credentials, raw MCP results, upstream bodies, and stack traces never cross the API boundary.
- Fixture mode remains deterministic, connected by default, and network-free.

## Risks and controls

- The current MCP client release is alpha. Pin it exactly and hide SDK-specific types behind the gateway adapter.
- Pref's provider catalog can change independently. Resolve exact tool refs during connection and mark mappings invalid instead of executing on drift.
- The hosted service requires a credential before discovery. Missing credentials must become `auth_required` without a network request; do not register an agent automatically.
- Native resource and prompt capabilities may be absent. Helper fallbacks retain only strict descriptor fields and enforce byte/count limits.
- MCP error messages can contain URLs or echoed arguments. Convert them to fixed safe codes/messages before recording or returning them.

## Results

- Inspected the active Pref MCP registration and live read-only helper surface without printing configuration values, headers, tokens, or claim links. Confirmed the hosted Streamable HTTP `/mcp` deployment, bearer/OAuth boundary, ten advertised helper tools, five public resources, one resource template, three prompt templates, and a 547-capability catalog snapshot.
- Reviewed the matching local Preference deployment source and pinned the official `@modelcontextprotocol/client` dependency to its exact `2.0.0-alpha.3` version. The SDK owns initialization, negotiated protocol version, session IDs, reconnect behavior, and best-effort session termination.
- Added `config/pref-capabilities.json` as the versioned source of provider names. Gameplay code knows only canonical capabilities; strict parsing enforces HTTPS `/mcp`, the `pref.trade` host allow-list, unique direct/provider tool grants, approved input projections, and read-only/non-destructive annotations.
- Selected `weather.get_current_conditions` on `weather_toolkit` for `local_conditions`. Exact `search_tools` lookup must still match provider server, strict required input fields, safety annotations, and read-only side-effect metadata before the mapping becomes executable.
- Rejected the discovered aggregate-news candidate because its current security metadata reports an external-write side effect and forbids task execution.
- Implemented an official Streamable HTTP connection adapter with server-only token providers, explicit disconnected/connecting/connected/auth-required/error transitions, bounded pagination, native resource/prompt discovery, Pref helper fallbacks, fixed safe errors, and remote-close/reconnect handling.
- Added a streaming fetch boundary that rejects oversized declared or chunked response bodies before the MCP SDK parses them, followed by a second retained-payload size check and primitive count/page ceilings.
- Added fail-closed provider dispatch. Only a provider ref that appears in the checked-in allow-list and has a currently valid mapping can pass through `call_tool`; arbitrary direct tools and provider refs never reach the client.
- Added fixture/live runtime construction in the orchestrator. Fixture is deterministic, connected by default, and network-free. Live mode reads only `SIGNAL_ATLAS_PREF_URL` and `SIGNAL_ATLAS_PREF_BEARER_TOKEN` on the server; a missing token becomes `auth_required` without constructing a network client.
- Added safe `GET /api/runtime/pref`, `POST /api/runtime/pref/test`, and `POST /api/runtime/pref/disconnect` endpoints. Responses expose only connection state, credential presence, endpoint host, bounded primitive summaries, mapping status, versions/counts, and fixed errors.
- Added a Pref MCP panel to the runtime diagnostics UI with status, fixture/live/auth boundary, discovered primitive names, mapping validity, disconnect, and test/reconnect controls. The Agent Dock's existing Pref indicator now reflects changes made in the panel.
- Seeded tests with credential-shaped upstream errors and a helper `claim_link` value. Neither appears in connection diagnostics, endpoint bodies, or the rendered UI contract.
- Manually inspected the completed panel at 1440 × 900 after the browser journey passed; the Pref inventory and controls remain legible without obscuring the existing driver and scheduler information.

## Remaining risks

- The application live adapter was not given a standalone Pref bearer during this task. Actual Pref discovery was verified through the already-connected Codex MCP surface, while the application handshake, auth-required path, and complete protocol lifecycle were tested at the SDK boundary. A bearer must be supplied explicitly through `SIGNAL_ATLAS_PREF_BEARER_TOKEN` for the P5-003 live application smoke; the Codex OAuth credential will not be copied and no agent will be registered automatically.
- The official MCP client is still an alpha release. It is pinned exactly and isolated behind `PrefMcpSdkClient`; upgrades require a focused transport/session regression pass.
- The response cap applies independently to each HTTP response, including a long-lived SSE stream. A future notification-heavy deployment may need a separate bounded rolling policy instead of the current per-response ceiling.
- Discovery metadata is process-memory state. Durable source/cache/version behavior belongs to P5-003 and must still honor source rights before retaining any provider result.
