# P2-001 Worklog: Mission commands and queue

## Status

Complete.

## Goal

Turn a player objective into an explicit, reviewable mission draft and append mission state only after the orchestrator validates a confirmed command.

## Intended files

- `apps/orchestrator`: authoritative in-memory expedition runtime, fixture mission interpreter, and local HTTP endpoints.
- `apps/web/src/world-shell`: runtime client, editable confirmation form, dynamic projection model, and queue controls.
- `packages/simulation`: reused without UI shortcuts for command validation, idempotency tracking, event folding, and projection rules.
- `tests/e2e/mission-flow.spec.ts`: complete draft, confirmation, refresh, ambiguity, and cancellation journey.

## Boundaries

- Natural-language interpretation creates only a local draft. It cannot append an event or infer through unresolved ambiguity.
- React never marks a mission queued, reordered, or canceled optimistically. It refreshes the authoritative projection after an accepted command.
- The fixture interpreter is an orchestrator adapter; no component branches on fixture result keys or scripted evidence.
- This milestone queues missions but does not begin travel. Route selection and simulation scheduling belong to P2-002.

## Acceptance criteria

- Mira can be dispatched to Galehaven Weather Tower in two interactions: Dispatch, then Confirm mission.
- Ambiguous commands remain editable drafts with confirmation disabled.
- An exact duplicate command returns its original result and appends no events; conflicting idempotency-key reuse fails closed.
- The queue supports cancel and per-agent reorder commands.
- A queued mission survives browser refresh because the frontend reloads the orchestrator snapshot.
- The complete flow is keyboard operable.

## Results

- Added an `ExpeditionRuntime` that owns the current projection, append-only event list, command ledger, and stored accepted results. Supported commands are translated to strict `WorldEvent` envelopes and folded atomically through the pure reducer.
- Exposed snapshot, incremental event, draft interpretation, and command endpoints under the expedition resource. Invalid commands return structured validation issues, new commands return `202`, and exact retries return `200` with `duplicate: true`.
- Added a deliberately bounded Helios-3 fixture interpreter. It resolves known agent names, authored location aliases, and supported mission verbs while reporting every unresolved field instead of guessing.
- Replaced the placeholder command expansion with an editable mission confirmation form. Agent, destination, action, and objective remain view state until the player confirms them.
- Derived the agent dock and mission queue from refreshed `WorldProjection` data. Queue cancel and arrow controls submit domain commands; completed/canceled items leave the active list but remain in event history.
- Changed Playwright startup to launch the full local stack so browser tests exercise the same `/api` proxy and orchestrator used by `pnpm dev`.
- Reviewed the expanded 1440 × 900 mission draft and two-item queue. Controls remain legible, the world stays visible, and queue priority is expressed without relying on color.

## Remaining risks

- Runtime state is process-local at this milestone. It survives browser refresh but not an orchestrator restart; committed persistence arrives with the later replay/resilience work.
- HTTP snapshot refresh follows each mutation. A live event channel and reconnect cursor are still required by P2-002/P6-002.
- The fixture parser is intentionally narrow. It exposes uncertainty safely, but a future richer parser must preserve the same draft-only boundary.
