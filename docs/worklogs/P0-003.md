# P0-003 Worklog: Domain contracts and fixture

## Status

Complete.

## Goal

Create the strict, infrastructure-independent domain boundary for Signal Atlas and validate the supplied Helios-3 fixture with cross-record invariants.

## Intended files

- `packages/contracts/src`: Zod schemas and inferred TypeScript types for the full domain.
- `packages/contracts/test`: valid fixture, invalid probability, missing-source, identity, and range tests.
- `packages/contracts/scripts`: deterministic JSON Schema generation.
- `schemas/*.schema.json`: generated or synchronized exported schemas.
- `packages/test-fixtures`: typed fixture loader with no UI or infrastructure dependency.
- `docs/OPEN_QUESTIONS.md`: resolution notes for contract ambiguities found in the design handoff.

## Constraints

- `@signal-atlas/contracts` must not import React, Phaser, Fastify, Codex, MCP, persistence, or UI packages.
- The supplied fixture remains byte-for-byte unchanged unless a documented defect makes that impossible.
- Binary probabilities must have exactly the market outcome keys and sum to one within a strict tolerance.
- Sources, claims, signals, routes, places, agents, Professor evidence, scripted results, and events require referential integrity.
- World events and commands use discriminated unions rather than arbitrary payloads.

## Acceptance criteria

- The Helios-3 fixture validates through the implementation contract.
- Invalid probability distributions fail with a precise path and message.
- Claims/signals with missing or dangling source IDs fail clearly.
- TypeScript types are inferred from the schemas and exported from the package.
- JSON Schema artifacts are generated deterministically and have no drift.
- Contracts remain dependency-clean.
- Repository typecheck, lint, tests, and build pass.

## Validation commands

```text
pnpm --filter @signal-atlas/contracts generate:schemas
pnpm --filter @signal-atlas/contracts test
pnpm typecheck
pnpm lint
pnpm test
pnpm build
```

## Dependencies and risks

- The original fixture contains nullable optional timestamps; the contract must preserve compatibility while normalizing future authoring guidance.
- The fixture is a content catalog plus two initial events, not yet a complete replay stream; P0-004 must ensure catalog signals do not become active before events record them.
- Event payload coverage must be strict enough for an exhaustive reducer without over-designing later live integrations.

## Results

- Added strict Zod contracts and inferred TypeScript types for markets, manifests, agents, missions, expeditions, sources, claims, signals, knowledge edges, beliefs, meetings, Professor queries/responses, agent turns, commands, events, and the complete expedition fixture.
- Defined ten closed mission verbs, seven Professor modes, ten command variants, and forty-five event variants. Event and command payloads are discriminated unions with versioned envelopes rather than arbitrary records.
- Added fixture-level referential validation for outcome keys, routes, places, agents, sources, claims, signals, Professor evidence, scripted mission keys/results, event identities/sequences, expedition cursors, and resolution outcomes.
- Added source-grounding validation inside agent-turn output: proposed claims and signals may cite only declared sources, signal claim indexes must exist, and signals must retain the sources used by their claims.
- Kept `fixtures/helios3_expedition.json` byte-for-byte unchanged and exposed it through `@signal-atlas/test-fixtures` as a validated canonical value plus isolated clone factory.
- Generated eighteen deterministic Draft 2020-12 JSON Schema artifacts from the Zod source and added a byte-for-byte drift check. Runtime Zod refinements remain authoritative for sums and cross-record invariants that JSON Schema cannot represent directly.
- Resolved Professor-mode, runtime-event, agent-action, and source-location ambiguities in `docs/OPEN_QUESTIONS.md`.
- Added thirteen contract tests and one typed-loader test covering the canonical fixture plus invalid probability, outcome count, range, source, identity, route, event sequence/version, command actor/type, scripted result, Professor evidence, and agent-turn grounding cases.
