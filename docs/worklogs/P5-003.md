# P5-003 Worklog: First live Pref mission

## Status

Complete.

## Goal

Turn the approved real Pref weather result into canonical source, claim, signal, dialogue, and agent-turn records while preserving the deterministic fixture path, explicit provenance, safe cache behavior, and the fictional boundary of the Helios-3 market.

## Pre-implementation evidence

- Exact live execution of `weather.get_current_conditions` for `Cape Canaveral, Florida` returned MCP `structuredContent` plus a text JSON compatibility copy.
- The observed envelope contained geocoded name/latitude/longitude, temperature, humidity, wind speed/direction, weather code/description/category, pressure, a provider observation `timestamp`, and a Unix-second `retrieved_at`.
- The matching Preference source defines `timestamp` as the Open-Meteo observation time and `retrieved_at` as toolkit retrieval time. Signal Atlas therefore records its own gateway retrieval separately instead of collapsing these meanings.
- Helios-3, Galehaven, and their market are fictional. Real Cape Canaveral conditions cannot truthfully become directional evidence for that market.
- No standalone Signal Atlas bearer was available. The real read-only result was inspected through the existing Pref connector; the application transport and end-to-end mission path use the same validated contract with a recorded rights-safe envelope in automated tests.

## Intended files

- `packages/pref-gateway/src/live-gateway.ts` and tests: strict live payload validation, canonical evidence, rights-aware source normalization, caching, versions, and stale fallback.
- `packages/pref-gateway/src/mcp-connection.ts` and package metadata: bounded Streamable HTTP execution that honors the owner's environment proxy.

## Acceptance criteria

- An approved live weather envelope produces one linked source, claim, signal, dialogue line, and completed agent turn.
- The source exposes the exact `weather.get_current_conditions` primitive, server, transport, call ID, argument hash, response hash, content hash, external identifier, version, and predecessor when applicable.
- Observation, provider retrieval, and gateway retrieval remain semantically separate. Publication time stays absent when the provider did not supply it.
- Fresh cache hits avoid a provider call; changed content creates a new source version with `supersedesSourceId`; retryable upstream failure may reuse only the last validated result and must label the resulting signal stale.
- Live Cape Canaveral data is visibly identified as a real-world proxy, produces only `context` direction and `unknown` impact, and does not update the fictional market belief.
- Fixture mode remains the default, deterministic, credential-free, and network-free path.
- Fixture/live selection and all credentials remain server-side. Unknown or changed provider tools still fail closed.

## Risks and controls

- **Fiction laundering:** A plausible real location could be mistaken for Galehaven. The capability configuration, source title/tags/location, claim qualifiers, signal headline/summary, dialogue, inspector banner, and unknowns all repeat the proxy boundary.
- **Timestamp fabrication:** The provider exposes no publication time. The adapter leaves it absent, parses the observation time as UTC according to the toolkit's default Open-Meteo request behavior, retains provider retrieval as typed evidence, and stamps gateway retrieval independently.
- **Unasserted display rights:** The raw provider payload is used transiently to form bounded factual claims but is not persisted in `SourceRecord`. Rights default to `metadata_only` and the content cache retains only canonical in-memory evidence/source data.
- **Stale data laundering:** Only retryable connection/upstream errors can use a prior validated cache entry. Cancellation, mapping drift, denied tools, invalid requests, and invalid payloads fail instead. Cache-derived claims/signals and the inspector are explicitly stale.
- **Proxy bypass:** Node 22 global fetch does not automatically consume the shell proxy variables. The official MCP client now uses Undici's environment proxy dispatcher and closes it with the MCP session.
- **Credential leakage:** Only the orchestrator reads the bearer. Public diagnostics expose configured/missing state, not values; tests scan HTTP bodies for credential-shaped terms and upstream details.

## Results

- Added a live canonical gateway for the single approved mapping. It validates `structuredContent` first, accepts bounded text JSON only as a compatibility fallback, and rejects extra/missing/wrong-typed weather fields.
- Normalized live weather into metadata-only sensor sources with content-addressed identities and the exact provider primitive. The provider retrieval field remains typed transient evidence; observation and gateway retrieval occupy their canonical source fields; publication remains unrecorded.
- Added a five-minute in-memory fresh cache, configurable at construction for tests. Identical fresh requests return the immutable validated source without another provider call. Changed canonical content increments the version and links the previous source. Retryable failure after expiry returns the prior source with `cache.status=stale` and a fixed warning.
- Added the `pref_proxy` driver kind and routed it through the existing bounded scheduler. Only a local-conditions observation mission is intercepted; all other missions delegate to the configured local Codex or scripted driver.
- Authored a Cape Canaveral capability configuration on Galehaven Weather Tower. The proxy materializer creates linked canonical objects while enforcing `direction=context`, `impact=unknown`, no outcome target/range, unverified proxy reliability, and explicit fictional-place qualifiers.
- Prevented context-only signals from generating belief updates. Added source supersession events, automatic staling of signals linked to an older source version, and a distinct stale-cache signal so the audit trail retains both the original and degraded interpretation.
- Connected the live gateway to the Pref runtime and application construction. `SIGNAL_ATLAS_PREF_MODE` is the server-side fixture/live toggle; fixture remains the default and live mode requires an explicit bearer.
- Added an HTTP-layer recorded-live demonstration that traverses the real gateway adapter, agent proxy, runtime scheduler, reducer, and application endpoints and proves source, claim, signal, dialogue, exact primitive, unchanged belief, and credential-free responses.
- Enhanced runtime diagnostics to show canonical-to-provider mappings and the server-side mode lock. Enhanced the source inspector with real-world proxy, stale audit, source ID, predecessor, tag, and existing exact primitive/hash details.
- Added environment-proxy support to the live MCP transport so the required `source ~/clash.sh; proxy_on` variables control outbound traffic.

## Remaining risks

- Live application verification now covers the single approved weather mapping and Cape Canaveral proxy only. Credentials remain a local operator responsibility, are not copied from Codex, and must never enter tracked files, public diagnostics, world events, or exported case files.
- The cache is intentionally process-local and bounded to one latest result per canonical input. Restarting loses it; durable encrypted cache policy and eviction accounting are post-MVP work.
- Open-Meteo/Preference licensing terms should be reviewed before enabling persistent full-payload display. Until then, live sources remain metadata-only.
- `weather.get_current_conditions` is the only live provider capability. Any second capability requires a new allow-list entry, exact discovery validation, rights review, and a separate materializer.
