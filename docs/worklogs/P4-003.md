# P4-003 Worklog: Agent profiles and bounded knowledge packets

## Status

Complete.

## Goal

Give local Codex agents distinct, versioned public behavior without allowing style to alter evidence truth or permissions. Every turn must receive only its agent's known records, current-turn evidence, or an explicit archive grant; unsupported actions and overlong public output must fail validation. Per-agent Codex sessions must survive an orchestrator restart without carrying transient archive access into later turns.

## Intended files

- `packages/codex-runtime/src/profiles.ts`: exact versioned role profiles, action allow-lists, and public-output validation.
- `packages/codex-runtime/src/knowledge-packet.ts`: deterministic, compact, allow-list-based source and signal packets.
- `packages/codex-runtime/src/session-registry.ts`: in-memory and append-only JSONL session registries.
- `packages/codex-runtime/src/prompt.ts`: profile and knowledge packet serialization into the bounded turn prompt.
- `packages/codex-runtime/src/local-exec-driver.ts`: packet-reference checks, profile enforcement, and compatible durable resume.
- `apps/orchestrator/src/local-fixture-codex-driver.ts`: fixture profile selection, archive-grant policy, and durable local runtime configuration.
- `apps/orchestrator/src/expedition-runtime.ts`: public rationale, unknowns, and profile identity propagation into authoritative events.
- `packages/contracts/src/events.ts` and generated schemas: replayable completed-turn audit fields.
- `packages/simulation/src/state.ts` and `reducer.ts`: projection of the completed-turn audit fields.

## Boundaries

- Profiles are resolved by exact `(role, profileVersion)`. Unknown versions fail closed instead of silently inheriting current behavior.
- Each profile supplies public style, an explicit action-type allow-list, and hard public dialogue/rationale/unknown limits. Validation is code-level and runs after canonical output parsing; it is not prompt-only guidance.
- Public dialogue is limited to 240 characters and one paragraph. Every accepted turn must state at least one material unknown. Public rationale is capped at 320 characters.
- The packet builder accepts the complete fixture collections only as lookup inputs. It emits records selected by known source/signal IDs, current-turn source IDs, or a separately labeled archive grant.
- Current-turn source access does not imply access to a pre-authored signal. Signal summaries appear only through an existing knowledge edge or an explicit archive grant.
- Archive access requires both the Archive Quarter archetype and an archive mission verb (`search_history` or `compare_sources`). The grant lists its place, verb, source IDs, and signal IDs in the packet for inspection.
- The driver validates every output reference against records actually included in the bounded packet, including packet size truncation, rather than trusting the model or the broader input ID arrays.
- Role style lives beside the evidence packet. Changing a profile changes behavioral instructions and permissions but does not clone, summarize differently, or mutate source truth.
- Completed-turn events persist the exact profile ID/version, public rationale, and unknowns. Replaying events therefore preserves the public audit explanation rather than only source and signal IDs.
- Session registry keys include expedition and agent identity. Resume occurs only when the stored profile ID and version match the active profile.
- Any turn with a transient archive grant starts without a prior session and does not persist its returned session. This prevents unused archive context from leaking through conversation memory into a later ordinary mission.
- The production local orchestrator defaults to `~/.local/state/signal-atlas/codex-runtime`; tests may inject an in-memory registry or isolated runtime root. Registry files are mode `0600` and contain no prompts, output text, credentials, or private reasoning.
- Scripted fixture mode remains the default and needs no filesystem state or local Codex installation.

## Acceptance criteria

- A weather mission for Mira contains the current weather source and no archive signal.
- An Archive Quarter history mission for Orin receives an explicit grant containing the authored archive source and base-rate signal.
- The same evidence packet is byte-for-byte structurally equal when only the active role style changes.
- A Scout output attempting the Analyst-only `update_belief` action is rejected, repaired once, and reduced to a safe wait if repeated.
- Overlong or multi-paragraph dialogue and missing unknowns are rejected independently of the model prompt.
- A completed turn replays with profile identity, concise public rationale, and explicit unknowns.
- A second driver and registry instance resumes the saved agent session after restart.
- Transient archive sessions are not saved.
- Real local Codex completes Mira's mission both before and after a process restart, with the second invocation using `codex exec resume` and no scripted fallback.

## Remaining risks

- The vertical-slice archive grant selects authored fixture records by archive source class. P5 must construct grants from audited Pref capability results, not from a broad backing collection.
- JSONL session storage is append-only. A long-running production service will need safe compaction or rotation, although each line is intentionally small and contains only identifiers and timestamps.
- Resumed non-archive sessions retain prior ordinary turns. This is compatible with the current monotonic knowledge-edge model; any future knowledge revocation feature must invalidate or scope sessions by an access-set revision.
- Model latency, availability, and behavior remain nondeterministic. Deterministic scripted mode stays the required offline and CI path.
