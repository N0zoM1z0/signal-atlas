# P4-001 Worklog: Replaceable Codex runtime boundary

## Status

Complete.

## Goal

Introduce one typed boundary for every game-agent turn before any local process integration is allowed into the application. The offline fixture must use that same boundary, while the runtime owns bounded concurrency, cancellation, timeout, persistence, validation, and safe operator diagnostics.

## Intended files

- `packages/contracts/src/agent-turn.ts` and `schemas/agent-turn-input.schema.json`: the complete turn packet and generated JSON Schema.
- `packages/codex-runtime/src/types.ts`: driver, result, diagnostics, event, error, and persisted-record contracts.
- `packages/codex-runtime/src/scripted-driver.ts`: deterministic implementation of `CodexDriver`.
- `packages/codex-runtime/src/scheduler.ts`: FIFO queue, configurable concurrency, cancellation, timeout, output validation, and terminal events.
- `packages/codex-runtime/src/store.ts`: in-memory and append-only JSONL turn stores.
- `apps/orchestrator/src/fixture-mission-driver.ts`: authored mission behavior adapted through `CodexDriver`.
- `apps/orchestrator/src/expedition-runtime.ts` and `app.ts`: injected driver boundary and diagnostics endpoint.
- `apps/web/src/world-shell/RuntimeDiagnosticsDialog.tsx`: privacy-safe runtime inspection UI.
- `tests/e2e/runtime-diagnostics.spec.ts`: the fixture mission-to-diagnostics browser journey.

## Boundaries

- `AgentTurnInput` carries only a validated mission, effective destination, bounded capabilities, known source/signal IDs, attempt identity, request time, and timeout. It carries no repository path, credential, or private reasoning.
- Scripted mission selection is reachable from the game runtime only through `CodexDriver.runTurn`; the fixture adapter remains independently testable, but the orchestrator no longer invokes it directly.
- A driver may be synchronous or asynchronous. The generic scheduler normalizes both forms and validates the final output again at the runtime boundary.
- Output agent and mission identities must match the scheduled input. A mismatch fails closed before any result can enter authoritative world state.
- The scheduler owns FIFO admission and limits active turns independently of driver behavior. Its default is two concurrent turns and is configurable with a positive integer.
- Cancellation and timeout are terminal persisted states with distinct error codes and explicit `turn.canceled` and `turn.timed_out` runtime events.
- An abort wins the scheduler race even if a faulty driver ignores its signal. A future process driver is still responsible for killing its subprocess when that signal fires.
- Runtime persistence stores metadata and terminal outcomes, not prompts, source bodies, credentials, stdout, stderr, or hidden reasoning. JSONL is append-only with mode `0600`; the latest valid record wins on reload.
- A restart converts abandoned queued/running records into an explicit recoverable failure rather than pretending the work completed.
- Diagnostics expose driver identity, mode, availability, run count, queue capacity, status totals, bounded turn metadata, and sanitized events only.
- No application or game-scene service imports child-process code. Process ownership is reserved for the runtime package in P4-002.

## Acceptance criteria

- `CodexDriver` is generic over validated turn input and driver artifacts and reports safe diagnostics.
- The deterministic Helios-3 fixture implements the same interface that the local Codex driver will implement.
- Generated and runtime schemas validate the complete agent-turn input and output boundary.
- Two turns run concurrently when configured with two slots; a third remains queued until one slot is released.
- Queued and running turns can be canceled, each producing a terminal record and explicit event.
- A turn exceeding its deadline becomes `timed_out` even when its driver never resolves.
- Completed, failed, canceled, and timed-out turns are persisted and included in diagnostics.
- The orchestrator serves diagnostics without prompt text, source content, private reasoning, auth paths, or secrets.
- The Agent Dock opens a keyboard-accessible diagnostics dialog that makes the scripted driver, scheduler capacity, and persisted outcomes legible.
- Dispatching Mira's authored weather mission increments the driver run count and produces one completed turn in the browser-visible diagnostics.

## Remaining risks

- Expedition Runtime remains synchronous while using the deterministic fixture. P4-002 integrates the asynchronous scheduler with local `codex exec`, process termination, repair, and scripted fallback.
- JSONL is a durable local audit format, not a multi-process transactional database. The vertical slice runs one orchestrator process, so concurrent writers are deliberately out of scope.
- Scheduler events currently live in bounded process memory while turn outcomes are durable. P6 can persist the sanitized event stream if replay requirements expand to runtime-operational events.
